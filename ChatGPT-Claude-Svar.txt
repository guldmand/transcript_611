## Et vigtigt problem at flagge

Der er **ét issue** jeg lige opdager. V2's baseline-implementationer konstruerer output-path internt med en hardcoded strategi-folder. Lad mig verificere:**Aha — her er issue'et.** V2's `sma_crossover` baseline-implementation skriver til den hardcoded path `sma_crossover/`, ikke til en parametrisk path som `sma_50_200/`.

Det betyder:
- Hvis grid runner kalder `run_sma_crossover(fast_window=50, slow_window=200)` → output går til `algorithmic_baselines/demo_10_new/sma_crossover/ko/`
- Hvis grid runner derefter kalder `run_sma_crossover(fast_window=20, slow_window=50)` → output **overskriver** den forrige fordi det skriver til samme `sma_crossover/ko/`

## Konsekvens — vi har to options

**Option A: Tilføj `strategy_folder_override` parameter til hver baseline-funktion**

Lille refactor af 3a-3d baselines hvor `run_*` accepterer en optional `strategy_folder` parameter:

```python
def run_sma_crossover(
    *,
    trade_data,
    dataset_tag,
    ticker,
    initial_amount=1_000_000.0,
    fast_window=50,
    slow_window=200,
    transaction_cost_pct=0.0,
    strategy_folder: str | None = None,  # NEW — overrides default "sma_crossover"
):
```

Hvis `strategy_folder=None`, brug default (`"sma_crossover"`) — bevarer 3a-3d backward compatibility.
Hvis `strategy_folder="sma_50_200"`, brug den i stedet.

**Pro:** Grid runner kalder `run_sma_crossover(..., strategy_folder="sma_50_200")` og får korrekt path. Default-konfigurationer fra etape 3a-3d virker stadig uden ændringer.

**Con:** Vi rører 3a-3d baseline-filerne. Men ændringen er minimal (én optional parameter med default `None`).

**Option B: Grid runner kopierer outputs efter hver kørsel til parametrisk folder**

Grid runner kalder `run_sma_crossover()`, lader den skrive til `sma_crossover/ko/`, og kopierer/renamer derefter folderen til `sma_50_200/ko/`.

**Pro:** Ingen refactor af 3a-3d baselines.

**Con:** Hacky. Hvis to konfigurationer kører parallelt (eller hvis runner crasher midt i), kan filer blive korrupt eller overskrevet. Race conditions.

## Min anbefaling: Option A

Option A er den **rene løsning**. Refactoren er minimal:

1. Tilføj `strategy_folder: str | None = None` keyword parameter til hver `run_*` funktion
2. Hvis `strategy_folder is None`, brug default (eksisterende string)
3. Hvis `strategy_folder` is set, brug den i stedet for default

Det er en **fully backward-compatible additive change**:
- Eksisterende kald som `run_sma_crossover(...)` virker uændret
- Etape 3a-3d outputs forbliver under `sma_crossover/`
- Grid runner kan eksplicit specificere `strategy_folder="sma_50_200"` for parametriske kørsler

## DeMiguel paperets pointe

**DeMiguel, Garlappi, Uppal (2009)** — "Optimal Versus Naive Diversification: How Inefficient is the 1/N Portfolio Strategy?"

Paperet viser at en simpel **1/N strategi** (lige store vægte til alle assets, rebalanseret periodisk) typisk slår sofistikerede mean-variance optimization strategier ud-af-sample. Dette er en **central baseline** i porteføljeteori.

Der er typisk **to versioner** af 1/N:

1. **1/N buy-and-hold** (statisk) — Allokér 1/N i starten, hold uden rebalansering. **Det er præcis hvad V1's `equal_weight_buy_and_hold_pit_portfolio` gør.**

2. **1/N rebalanced** (dynamisk) — Allokér 1/N månedligt/kvartalsvis ved at sælge winners og købe losers. Dette er den "canonical DeMiguel 1/N" version.

V1 har version 1, men **ikke** version 2.

    *,
    trade_data: str | Path,
    dataset_tag: str,
    initial_amount: float = 1_000_000.0,
    rebalance_frequency_days: int = 21,  # default monthly

) -> Dict[str, Path]

Algorithm:

2. Drop tickers with incomplete histories.
3. Set initial allocation: 1/N capital per asset on first trade date.
4. On every rebalance_frequency_days bar, compute current weights from 
   current values. If any weight deviates from 1/N, rebalance by selling 
   over-weighted assets and buying under-weighted assets to restore 1/N.
5. Track portfolio value daily.

PIT safety:
- Rebalancing decisions made on day t use only data available through day t-1 
  (apply weights at day t close based on day t-1 prices).
- This matches the PIT convention used by all etape 3 baselines.
