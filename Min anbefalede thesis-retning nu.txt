## Hvad betyder action space?

I reinforcement learning betyder **action space**:

> Hvilke handlinger agenten må vælge mellem på hvert tidspunkt.

Det handler ikke om datatypen på inputvariablerne, men om **agentens mulige beslutninger**.

## Discrete action space

Et **discrete action space** betyder, at agenten vælger én handling fra en fast liste.

Eksempel fra CartPole:

```text
0 = skub til venstre
1 = skub til højre
```

Eksempel i trading:

```text
0 = CASH
1 = køb/vælg AAPL
2 = køb/vælg MSFT
3 = køb/vælg NVDA
4 = køb/vælg AMZN
```

Eller:

```text
0 = hold
1 = buy
2 = sell
```

Eller mere DSS-agtigt:

```text
0 = hold current portfolio
1 = rebalance conservative template
2 = rebalance moderate template
3 = rebalance aggressive template
4 = reduce risk
5 = increase risk
```

Det er diskret, fordi agenten vælger mellem **klart adskilte muligheder**.

IQN, DQN og QR-DQN passer naturligt til denne type.

## Continuous action space

Et **continuous action space** betyder, at agenten vælger tal på en glidende skala.

I FinRL stock trading kan handlingen typisk være en vektor med én værdi per aktie:

```text
AAPL:  0.37
MSFT: -0.12
NVDA:  0.85
...
```

eller mere praktisk:

```text
AAPL: køb 37 aktier
MSFT: sælg 12 aktier
NVDA: køb 85 aktier
```

Det er continuous, fordi handlingen ikke bare er “vælg mulighed 0, 1 eller 2”. Den kan være mange mulige numeriske værdier.

Det er derfor algoritmer som:

```text
DDPG
TD3
SAC
```

bruges i FinRL-tutoriallen. De er gode til continuous action spaces.

I dit FinRL-script bliver `StockTradingEnv` oprettet med:

```python
"action_space": stock_dimension
```

og `stock_dimension` er antallet af aktier i datasættet. Det betyder, at miljøet har én handlingsdimension per aktie. Træningsscriptet bruger derefter A2C, DDPG, PPO, TD3 og SAC via FinRL/Stable-Baselines3. 

Så FinRL-standardmiljøet er ikke “vælg én aktie”. Det er mere:

```text
vælg køb/salg/positionering for alle aktier samtidig
```

## Din plante/alder-forståelse

Du er tæt på, men i RL-sammenhæng skal du tænke sådan:

```text
Continuous:
agenten kan vælge et tal på en skala
fx køb 13.72%, køb 14.01%, køb 14.02% osv.

Discrete:
agenten vælger fra en endelig liste
fx CASH, AAPL, MSFT, NVDA
```

Din plante-eksempel er fint for continuous data:

```text
planten gror 12.37 cm
```

Alder som `[0–100]` kan være diskret, hvis du kun tæller hele år:

```text
0, 1, 2, 3, ..., 100
```

men continuous, hvis du måler præcist:

```text
37.64 år
```

Men i RL handler det altså om **hvilke handlinger agenten kan tage**, ikke om hvorvidt inputdata er continuous eller discrete.


## Hvorfor A er bedst for din egen IQN

IQN er grundlæggende en DQN-lignende algoritme. Den estimerer værdien af hver mulig diskret handling:

```text
Q(state, action, tau)
```

Hvis handlingerne er:

```text
0 = CASH
1 = AAPL
2 = MSFT
3 = NVDA
```

så kan IQN outputte quantiles for hver action:

```text
AAPL: q10, q50, q90, CVaR
MSFT: q10, q50, q90, CVaR
NVDA: q10, q50, q90, CVaR
CASH: q10, q50, q90, CVaR
```

Det er perfekt til dit DSS-mål:

```text
Recommendation:
vælg MSFT

Reason:
MSFT has the best CVaR-adjusted score under the moderate risk profile.
q10 = ...
q50 = ...
q90 = ...
uncertainty = ...
```

Hvis vi prøver at lave IQN direkte på FinRLs continuous action space, skal agenten i stedet estimere en return distribution for uendeligt mange mulige porteføljevægte. Det er meget sværere og ikke længere klassisk IQN.

Hvad med continuous IQN?

Man kan godt lave noget, der ligner continuous IQN, men så er det ikke længere klassisk IQN på samme måde.

Du får typisk noget i retning af:

actor(state) → continuous portfolio weights
critic(state, action, tau) → distributional value estimate

Det ligner mere:

distributional actor-critic
SAC/TQC/DDPG-style system

end klassisk IQN.

Det kan være spændende, men det er sværere at forklare, sværere at validere, og større risiko for at drukne i implementation.

## Skal du kopiere FinRL-strukturen?

Ikke direkte.

FinRL-strukturen er god til:

```text
data download
feature engineering
StockTradingEnv
SB3 baseline training
backtesting
```

Men din IQN skal ikke ligne FinRLs SB3-wrapper. Den skal ligne en ren RL-agent:

```text
env.reset()
for step:
    action = agent.act(state)
    next_state, reward, done, info = env.step(action)
    replay_buffer.add(...)
    agent.learn()
```

Det er mere ObjectRL/DQN/IQN-style.

## Den rigtige progression

Jeg ville ikke starte med ObjectRL-A2C/DDPG/PPO/TD3/SAC.

Det giver ikke så meget værdi, fordi du allerede har dem gennem FinRL/SB3. Din baseline er netop FinRL-standardflowet: data-script, training-script og backtest-script. README’en beskriver også, at tutoriallen træner 5 DRL agents med Stable-Baselines3 og gemmer modellerne i `trained_models/`. 

Så progressionen bør være:

## Stage 1 — FinRL baselines

Brug FinRL/SB3 som du gør nu:

```text
finrl_continuous_env.py
A2C
DDPG
PPO
TD3
SAC
MVO
DJIA
```

Her skal du ikke blande ObjectRL ind.

Formål:

```text
etabler klassiske baseline-resultater
```

## Stage 2 — Lav dit discrete environment

Byg:

```text
src/stockdss/envs/finrl_discrete_env.py
```

Dette miljø bruger samme data:

```text
train_data_50.csv
trade_data_50.csv
train_data_500.csv
trade_data_500.csv
```

men action space er diskret:

```text
0 = hold
1 = reduce risk
2 = increase risk
3 = move to cash
4 = rebalance conservative
5 = rebalance moderate
6 = rebalance aggressive
```

Formål:

```text
skab et miljø, hvor DQN/QR-DQN/IQN giver mening
```

## Stage 3 — Simpel sanity check

Før IQN: lav en meget simpel random eller rule-based agent på `finrl_discrete_env.py`.

Eksempel:

```text
Random agent
Always hold
Always conservative
```

Formål:

```text
sikre at miljøet virker før neural RL
```

## Stage 4 — QR-DQN baseline

Brug QR-DQN som distributional baseline på samme discrete env.

Formål:

```text
sammenlign IQN mod en kendt distributional DQN-variant
```

## Stage 5 — ObjectRL-style IQN

Byg din egen:

```text
src/stockdss/agents/iqn/
```

med:

```text
iqn_network.py
= PyTorch netværket: state + tau → quantile Q-values

replay_buffer.py
= experience replay

risk_policy.py
= mean/q10/q50/q90/CVaR action selection

iqn_agent.py
= act(), learn(), update_target()

train_iqn.py
= træningsloop
```

Formål:

```text
dit thesis contribution
```

## Stage 6 — DSS layer

Byg output ovenpå IQN:

```text
recommendation
risk profile
q10/q50/q90
CVaR
uncertainty
explanation
audit log
```

Det er her specialet bliver decision support og ikke bare trading.

## Bør du lave ObjectRL-eksempler på A2C/DDPG/PPO/TD3/SAC?

Mit svar: **nej, ikke som prioritet.**

Det vil give dig en tredje baseline-stack:

```text
FinRL/SB3 A2C/DDPG/PPO/TD3/SAC
ObjectRL A2C/DDPG/PPO/TD3/SAC
IQN custom
```

Det bliver for meget og forvirrer specialet.

Hold det rent:

```text
FinRL/SB3 = standard baselines
ObjectRL-style = din egen IQN implementation style
Dopamine = IQN reference implementation
```

## Skal du bruge `finrl_continuous_env.py` til ObjectRL?

Ikke nødvendigt i første omgang.

`finrl_continuous_env.py` er primært til FinRL/SB3-baselines.

Din ObjectRL-style IQN bør bruge:

```text
finrl_discrete_env.py
```

fordi klassisk IQN passer til diskrete actions.

## Den praktiske implementeringsplan

Start sådan:

```text
1. Behold FinRL baseline scripts
   data → train → backtest

2. Lav src/stockdss/envs/finrl_discrete_env.py

3. Test env med random agent

4. Lav src/stockdss/agents/iqn/replay_buffer.py

5. Lav src/stockdss/agents/iqn/iqn_network.py

6. Lav src/stockdss/agents/iqn/iqn_agent.py

7. Lav train_iqn.py

8. Lav backtest_iqn.py

9. Sammenlign mod FinRL baselines
```

## Kort svar

Du skal **ikke** rette i ObjectRLs filer.

Du skal **ikke** bygge ObjectRL-versioner af A2C/DDPG/PPO/TD3/SAC først.

Du skal gøre dette:

```text
FinRL/SB3:
bruges til continuous baselines

ObjectRL-inspired PyTorch:
bruges til din egen IQN

Dopamine:
bruges som reference for korrekt IQN-design

finrl_discrete_env.py:
bruges til QR-DQN og IQN
```

Det er den mest rene, realistiske og akademisk forsvarlige vej.

Min anbefaling

Brug dette som akademisk navn:

D-IQN-DSS

med fuld titel:

Discrete Implicit Quantile Network for Stock Investment Decision Support

Det passer godt, fordi det siger:

Discrete
= diskret action space

Implicit Quantile Network
= IQN-metoden

Stock Investment Decision Support
= dit specialeproblem
I kode kunne det hedde

I fil-/class-navne ville jeg bruge noget roligt og professionelt:

src/stockdss/agents/iqn/
    iqn_agent.py
    iqn_network.py
    replay_buffer.py
    risk_policy.py

Class-navne:

class DiscreteIQNAgent:
    ...
class IQNNetwork:
    ...
class RiskSensitivePolicy:
    ...

Environment:

class FinRLDiscreteEnv:
    ...

eller mere præcist:

class FinRLDiscreteDecisionEnv:
    ...
Undgå disse som akademiske hovednavne

Jeg ville undgå:

GuldmandIQN
CustomIQN
MyIQN
StockIQN
FinIQN

Ikke fordi de er “forkerte”, men fordi de enten lyder personlige, generiske eller uklare.

CustomIQN siger heller ikke hvad der er custom.

God navngivning i specialet

Du kan skrive:

The proposed method is referred to as D-IQN-DSS, a discrete-action Implicit Quantile Network adapted for stock investment decision support.

På dansk:

Den foreslåede metode betegnes D-IQN-DSS, en diskret-action Implicit Quantile Network-variant tilpasset beslutningsstøtte for aktieinvesteringer.

DSS-IQN

StockDSS-IQN

objectrl/
├─ agents/
